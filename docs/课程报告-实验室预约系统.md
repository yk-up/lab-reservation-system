## 四、技术选型与需求分析（对应分值：20 分）

### 4.1 需求概述

实验室预约系统面向 **普通用户** 与 **管理员**：用户浏览实验室、提交预约、查看通知与个人预约；管理员审批、维护实验室与黑名单、查看看板与统计等。

**非功能需求（摘要）：**

- 前后端分离、接口统一 JSON、鉴权（JWT）。  
- 首版部分能力明确「可先前端实现」（如天气占位、通知筛选、列表本地筛选），预留后端扩展字段与接口参数。

### 4.2 技术选型说明

| 层次 | 选型 | 选型理由（简要） |
|------|------|------------------|
| 前端框架 | Vue 3（Composition API） | 组合式 API 利于复杂页面状态与复用 |
| UI | Element Plus | 与 Vue 3 配套成熟，表格/表单/轮播等开箱即用 |
| 构建 | Vite | 开发时 HMR 快，适合课程实验节奏 |
| 请求 | Axios + 拦截器 | 统一 Token、错误提示与登录跳转 |
| 后端（若报告包含） | Spring Boot 3 + MyBatis | 企业主流栈，易与课程知识点对齐 |
| 数据库 | MySQL | 关系型数据适合预约、用户、公告等业务 |

### 4.3 需求到架构的对应（简图说明）

用文字描述即可：**浏览器** → **Vite 开发服务器（/api 代理至后端）** → **Spring Boot REST** → **MySQL**；静态资源与打包后部署方式可在一小段中说明。
